1(b)
Identify three reasons why it is important to enforce referential integrity.
[3]
✓ Mark Scheme Answer from MS
  • Improves data accuracy / reduces record anomalies / improves data retrieval accuracy
  • Improves data consistency / maintains relationships — ensures records in a linked table relate to data in another table (via PK and FK) to allow extraction from different tables
  • Ensures a deleted record does not leave hanging records unlinked to other tables
  • Simplifies update/maintenance through cascade delete/update
Accept related example: if a police officer who interviewed a witness is deleted, they cannot be identified from the interview record.
1(c)
Explain how a query can provide a view of the database.
[3]
✓ Mark Scheme Answer from MS

Cluster 1 — how a view is defined:

  • A name is assigned to the view: CREATE VIEW view_name AS
  • Fields (columns) are selected: SELECT field1, field2 FROM table(s)
  • A condition is specified: WHERE condition / filter applied
  • This reduces the subset of data to include only the required data

Cluster 2 — properties of a view:

  • A query/filter provides a subset of the database
  • It can be saved and used regularly (e.g. weekly updates)
  • The view is not part of the physical schema (virtual table)
  • The view can be organised/sorted according to reporting needs
Note: do not award marks from different clusters.
Question 2 Secondary key · Composite PK · 3NF · Data types · Normalisation
2(a)
Define the term secondary key.
[1]
✓ Model Answer
  • A field (or set of fields) used to index a table other than the primary key
  • Used to enable efficient searching/retrieval on non-primary-key columns
  • Does not need to be unique (unlike a primary key)
2(b)(i)
INTERVIEW(WitnessID, CrimeID, OfficerID, Statement, CrimeLocation, DateTimeStatement, PoliceStation, StationPhone) — composite PK = {WitnessID, CrimeID, OfficerID} Outline why the use of WitnessID, CrimeID and OfficerID is not a suitable composite primary key.
[2]
✓ Model Answer
  • A witness may be interviewed more than once by the same officer about the same crime
  • So the combination {WitnessID, CrimeID, OfficerID} would not be unique across all rows — duplicate combinations are possible
  • A primary key must uniquely identify every record; this composite key fails that requirement
  • Fix: add a surrogate InterviewID (auto-increment integer) as PK, or include DateTimeStatement in the composite
2(b)(ii)
Outline why the INTERVIEW table is not in 3rd Normal Form (3NF).
[4]
✓ Model Answer — 2NF and 3NF violations

2NF violation — partial dependency:

  • CrimeLocation depends only on CrimeID alone, not on the full composite key
  • 2NF requires every non-key attribute to depend on the whole primary key — this partial dependency breaks 2NF

3NF violation — transitive dependency:

  • StationPhone depends on PoliceStation, which is itself a non-key attribute
  • Dependency chain: PK → PoliceStation → StationPhone (transitive)
  • 3NF requires every non-key attribute to depend only on the primary key — this transitive dependency breaks 3NF
Resolution: move CrimeLocation to a CRIME table; create POLICESTATION(PoliceStation, StationPhone) and reference it by FK in INTERVIEW.
2(c)
Outline why an alphanumeric data type is used for the VictimPhone field.
[2]
✓ Model Answer
  • Phone numbers may have leading zeros (e.g. 01029…) — storing as an integer would strip them
  • Phone numbers may contain non-numeric characters: spaces, hyphens, +, country codes, brackets
  • No arithmetic is ever performed on a phone number — numeric type gives no advantage
  • Alphanumeric stores the number exactly as entered / as it would appear on a document

Question 3 Concurrency · Centralised DB · Data mining · Data matching · GDPR
3(a)
Two police officers attempt to update a witness statement at the same time. Describe how data locking can be used to deal with this situation.
[2]
✓ Model Answer
  • When Officer A begins editing the record, the DBMS places a lock on that record
  • Officer B's attempt to edit the same record is blocked / queued until the lock is released
  • Once Officer A commits/saves, the lock is released and Officer B can then access and make their update
  • This prevents a lost update anomaly — without locking, one officer's save would silently overwrite the other's
Row-level locking is preferable to table-level: other witnesses' records remain accessible during the lock.
3(b)
Discuss the advantages and disadvantages of using a centralised crime database.
[5]
✓ Model Answer — must be balanced for full marks

Advantages:

  • Single source of truth — all forces see the same up-to-date data; no version inconsistency
  • Cross-force searching — identify suspects / witnesses across regions without duplicate systems
  • Easier to maintain and secure — patches, backups, and audits applied in one place
  • Supports national crime statistics and resource allocation policy
  • Eliminates data duplication — more storage-efficient

Disadvantages:

  • Single point of failure — if the central server goes down, all forces lose access simultaneously
  • Network dependency — reliable high-bandwidth connectivity required at every station
  • High-value cyberattack target — a single breach exposes the entire national dataset
  • GDPR / data sovereignty complications when multiple jurisdictions share one system
  • Performance bottleneck under thousands of concurrent users
For full marks: at least 2 advantages AND 2 disadvantages must be developed, not merely listed.
3(c)(i)
Explain why data mining might be used to extract information from crime data held in a police database.
[4]
✓ Model Answer
  • Police databases contain millions of records — far too large for manual analysis
  • Data mining uses algorithms to discover hidden patterns, correlations, and trends that are not apparent from individual records
  • e.g. identifying crime hotspots — spatial/temporal clustering of crimes not obvious to human analysts
  • e.g. predicting repeat offender behaviour — patterns in past offences forecast future crimes, informing parole decisions
  • e.g. linking crimes from different stations sharing modus operandi or suspect descriptions
  • Allows police to allocate patrols and resources proactively rather than reactively
Mark requires: reason why manual analysis is insufficient + specific benefit illustrated with an example.
3(c)(ii)
Explain, using an example, how data matching could be applied to the crime data.
[4]
✓ Model Answer
  • Data matching compares records across two or more datasets to identify entries that refer to the same real-world entity
  • Can be exact (same ID field) or fuzzy (similar names, addresses — accounting for typos)

Examples in crime context:

  • Matching a suspect's name/DOB in the crime DB against the DVLA vehicle register to locate their current address
  • Matching witness phone numbers across separate crimes — ZY33 Zixin Yang appears in both crimes in Figure 1; matching reveals the same witness is linked to multiple incidents
  • Matching stolen goods descriptions against pawnbroker transaction records to trace receivers of stolen property
  • Cross-referencing fingerprint / DNA evidence across crime scenes to link crimes to a single perpetrator
Must include a clear, developed example for full marks. General description only → max [2].
3(d)(i)
Outline two reasons why a password-only approach would not be secure.
[4]
✓ Model Answer — 2 marks per reason: identify + develop
  • Passwords can be guessed / brute-forced: without account lockout, an attacker systematically tries combinations; weak or reused passwords are vulnerable to dictionary attacks
  • Passwords can be stolen / phished: officers may be tricked via fake login pages; credentials can be intercepted if transmitted without encryption
  • Passwords can be shared: officers sharing login details means individual access cannot be audited — the principle of least privilege is violated
  • Password does not verify physical identity: a stolen password grants full access; adding a second factor (smart card, biometric, OTP) ensures only the legitimate user can authenticate
3(d)(ii)
Identify three ways that legislation such as GDPR protects individuals whose data is stored electronically.
[3]
✓ Model Answer — 1 mark each
  • Right of access — individuals can request to see what data is held about them (Subject Access Request)
  • Right to erasure ("right to be forgotten") — data must be deleted if no longer necessary for its original purpose
  • Data minimisation — only data strictly necessary for the stated purpose may be collected and stored
  • Informed consent — explicit, freely-given consent must be obtained before collecting personal data
  • Data accuracy — organisations must keep data up to date; individuals can request corrections
  • Breach notification — individuals must be informed within 72 hours if their personal data is compromised
  • Right to portability — individuals can receive their data in a structured, machine-readable format
